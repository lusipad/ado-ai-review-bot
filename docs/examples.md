# 真实示例集

以下全部是开发期间在真实环境（Windows 11 + Azure DevOps Server 2022 + Rocket.Chat 8.0 + codex 0.144）实测产生的 bot 原始输出，未经修饰。

## 1. 深入 review：抓到「只看 diff 永远发现不了」的 bug

测试设计：PR 把 `applyDiscount` 的参数从「0~1 小数」改成「0~100 百分比」，**故意不改调用方**——diff 里只有 `discount.js`，问题却在没动过的 `checkout.js`。

bot 的行内评论（精确锚定在 `checkout.js:13`，该文件不在 diff 中）：

> **🔴 必须修复** 结算调用仍使用旧的小数折扣语义
>
> `applyDiscount` 现在要求传入 0~100 的百分比，但这里仍传入旧语义的 `0.05`。这会把注释声明的九五折解释成减免 0.05%：实际运行 `checkoutTotal([{ price: 10000, qty: 1 }])` 返回 9995，而九五折应返回 9500。应将参数改为 `5`，并补充覆盖 `checkoutTotal` 与 `applyDiscount` 百分比语义的回归测试；当前仓库没有任何测试文件，因此该调用迁移遗漏无法被自动发现。

注意它**实际算出了错误金额**（9995 vs 9500）——agent 在 worktree 里跑了代码验证，不是猜的。

> - `src/checkout.js:13`：需要确认业务规定的计税顺序是先折扣后计税，还是按原价计税，以及税率应从哪里取得。
> - `src/tax.js:4`：需确认退款场景是否允许负数 total；这会决定输入校验边界。
>
> 可以快速略过：严格模式声明和 CommonJS 导出属于现有模块的机械性约定。

## 4. @bot 线程问答（20 秒，带代码依据）

群聊提问：`@review-bot test/test 里的折扣是向下取整还是四舍五入？为什么？`

bot 在提问线程里回答：

> 折扣是**向下取整**。依据是 `src/discount.js` 的 `applyDiscount(price, rate)` 直接返回 `Math.floor(price * (1 - rate))`，注释也明确写了"折后价（分），向下取整"。`src/checkout.js` 会先对每件商品的单价调用该函数，再乘数量，所以取整发生在折后单价这一层；至于为什么选择向下取整，仓库里没有进一步说明，不能确定具体业务原因。

注意最后一句——查不到的事**直说**，不编造。

>

> - `items` 非数组时抛出 `TypeError('invalid items')`
> - 空数组直接返回 `0`
> - 保持原有单行校验风格，未修改无关代码
>
> 验证：仓库无现成测试配置；已通过 Node.js 断言验证非数组、空数组及正常结算场景。未执行任何 git 命令。
>
> _bot 将对该提交自动做一次增量 review。_

推送的代码与要求逐条对应；随后触发的增量 review 复核了它自己的修复。改动超护栏（默认 10 文件/300 行）或涉及 CI/评审配置时会拒绝并说明。
