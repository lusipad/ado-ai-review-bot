# 使用手册（开发者视角）

> 部署与配置见 [README](../README.md)；真实输出示例见 [examples.md](examples.md)。

## 一个 PR 的完整生命周期

```
建 PR（非草稿）
  └→ 自动全量 review（1~15 分钟，取决于变更规模与模型速度）
      ├─ 一条总评：摘要 · 变更导览 · 👀 给人工审阅者 · 整体风险 · 命令帮助
      ├─ 若干行内评论：🔴 必须修复 / 🟡 建议 / 🔵 细节（每轮默认最多 10 条）
      └─ PR status「ai-review」：pending → succeeded/failed

push 新提交
  └→ 防抖 3 分钟 → 增量 review（只审新变更）
      ├─ 已修复的旧问题：线程自动回复 ✅ 并置为 fixed
      └─ 新问题照常行内评论

PR 合并 / 放弃
  └→ 遗留未解决的意见归档为「带病合并」（进统计，不再出现在待处理清单）
```

**草稿 PR 不触发**，转正式时触发。**同一 commit 不会重复 review**（ADO 重发事件、防抖窗口内手动触发过都会被去重）。

## PR 里的交互

| 操作 | 写法 | 说明 |
|---|---|---|
| 问 bot 问题 | **新开评论**（行内或概览区）：`@<bot显示名> 这里为什么有风险？` | 1~2 分钟内先回「🔍 分析中」再编辑为答案；答案带代码引用 |
| 贴图提问 | 评论里贴截图 + `@bot 这是什么错？` | 图片一起交给模型（需模型支持视觉，最多 4 张） |
| 强制重审 | 评论 `/review` | 无视一切跳过条件，全量重审 |
| 让 bot 修 | 问题线程新开评论 `/fix` 或 `/fix 用早退代替嵌套` | bot 改代码 → push 到源分支 → 自动增量 review 自己的提交（需管理员开启，有规模护栏） |
| 跳过评审 | PR 标题或提交信息带 `[skip review]` | WIP/typo/格式化时省模型调用；`/review` 可随时强制 |

> ⚠️ **Server 2022 限制**：只有「新评论线程的首条评论」触发事件，**线程 Reply 框里的回复不触发**。@bot、/review、/fix 都要新开评论。

## 教 bot 学习（反馈闭环）

- **不认同某条意见**：在该线程回复一句理由（如「团队约定不强制注释」），把线程 Resolve 为 **Won't Fix** → bot 记住，同仓库不再提同类建议，理由会进入长期记忆参与每周整理；
- **认同并修复**：直接改代码 push，bot 会自动确认并关线程（或手动 Resolve）；
- **某仓库 nit 太吵**：什么都不用做——nit 采纳率持续过低会自动折叠；或在 `.ai-review.yml` 显式设 `minSeverity`；
- **审计 bot 学到了什么**：群聊发 `记忆 <项目/仓库>`，或直接编辑 `data/knowledge/<仓库>-memory.md`（删错的、补充口口相传的约定）。

## RocketChat 里的交互

触发词 `!review` 或 `@<bot用户名>` 开头（任何接入的频道）：

| 输入 | 行为 | 耗时 |
|---|---|---|
| `状态` | 队列实况：正在处理哪些 PR、排队/防抖 | 秒回 |
| `统计 [天数]` | review 次数、意见数、各仓库采纳率、带病合并 | 秒回 |
| `待处理` | 全部未解决 must-fix（带 PR 链接） | 秒回 |
| `架构 <项目/仓库>` | 仓库地图（模块/调用链/术语表） | 秒回 |
| `记忆 <项目/仓库>` | bot 积累的长期记忆 | 秒回 |
| 任意问题 | 进代码库核实后**在你消息的线程里**回答 | 1~2 分钟 |
| `讨论 <问题>` | 完整分析放进自动创建的讨论组 | 1~3 分钟 |
| `工作项 <id> [问题]` | 需求讨论组：工作项卡片 + 关联 PR 状态 + 实现影响面分析 | 1~3 分钟 |

自由提问的仓库定位：问题里写仓库名（`test/test`）> 频道默认绑定 > 唯一已知仓库。

## 通知

- review 完成 / 发现必修 / 任务失败会推送到群（must-fix @ 到 PR 作者，需配映射）；
- **静默时段**（如 21:00~09:00）通知积压，早上按仓库汇总一条——review 照跑，PR 上的结果不受影响；
- 周一 09:00 可选度量周报。

## 管理与观测

- 面板 `http://<bot>:<port>/admin`：概览卡片、队列、采纳率（含带病合并）、最近任务；
- `GET /stats?days=30&repo=...`（`x-webhook-secret` 头鉴权）程序化取数；
- 部署自检 `node dist/server.js --doctor`。

## FAQ

**Q：PR 建了但 bot 没反应？**
按序检查：是不是草稿？标题/提交带 `[skip review]`？变更是否全是图片/lockfile/忽略路径（status 会注明「已跳过」）？该仓库 `.ai-review.yml` 是否 `autoReview: false`？再看 ADO Service Hooks 订阅历史和 bot 日志。

**Q：@bot 没回我？**
是否用了已有线程的 Reply 框（不触发，要新开评论）？是否用 bot 自己的账号发的（自触发过滤）？

**Q：让 bot 只看重点、别碰某些目录？**
`.ai-review.yml`：`ignorePaths` 排除路径、`focus` 指定侧重点、`minSeverity` 提高门槛、`persona` 换语气。

**Q：review 太慢/太贵？**
增量机制已保证只审新变更；再省就用 `[skip review]`、调低 `REVIEW_PROFILES` 到单模型、`CODEX_EXTRA_ARGS=-c model_reasoning_effort=medium`。
