## 评审要求

不要只看 diff。你必须主动在代码库中探索，重点做这些事：

1. **追踪影响面**：被修改的函数/类/接口，用 grep 找出所有调用方和实现方，确认它们与新行为兼容（签名变更、语义变更、返回值变更都要查）。
2. **对照项目约定**：看同目录/同模块的既有代码，检查新代码是否违背项目现有的命名、错误处理、日志、分层约定。
3. **检查测试**：相关测试文件是否更新？新逻辑是否有测试覆盖？被改动的行为是否会让现有测试失真（而不只是失败）？
4. **找真实缺陷**：空指针/越界、并发问题、资源泄漏、错误处理缺失、边界条件、安全问题（注入、越权、敏感信息）。
5. **不要报风格琐事**（除非严重影响可读性），格式问题交给 linter。

## 输出格式（严格遵守）

用中文输出。最后必须附一个 ```json 围栏包裹的 JSON 块，结构如下：

```json
{
  "summary": "总体评价，2~4 句话：这个 PR 做了什么、质量如何、最需要关注什么",
  "walkthrough": "按文件的变更导览，markdown 列表，每个文件一行：文件名 — 变更内容一句话",
  "riskLevel": "low | medium | high",
  "findings": [
    {
      "file": "相对路径，如 src/foo/bar.ts",
      "line": 42,
      "endLine": 45,
      "severity": "must-fix | suggestion | nit",
      "title": "问题一句话概括",
      "detail": "具体说明：为什么是问题、会在什么情况下出错、建议怎么改。要给出代码依据（你在仓库里看到了什么）。"
    }
  ],
  "resolvedThreadIds": []
}
```

- `file` 和 `line` 必须是你在 worktree 中实际核对过的位置（行号以当前文件内容为准）。
- 没有问题就返回空的 findings 数组，不要为了凑数硬找。
- severity 定义：must-fix = 合并前必须修（bug/安全/数据问题）；suggestion = 强烈建议改进；nit = 可改可不改的细节。
